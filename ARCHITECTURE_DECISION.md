# アーキテクチャ決定記録 (ADR) - Backend 移行

## 1. 背景
現在、Java (Spring Boot) で構築されているバックエンドを AWS Lambda で運用するために、TypeScript への書き換えを決定しました。

## 2. 決定事項

### 2.1 言語・ランタイム
- **決定:** TypeScript (Node.js)
- **理由:** 
    - Lambda のコールドスタートが Java よりも高速。
    - メモリ消費が少なく、Lambda のコスト効率が良い。
    - AWS SDK v3 との親和性が高く、開発体験が良い。

### 2.2 データベース
- **決定:** Amazon DynamoDB
- **理由:**
    - サーバーレス（接続管理不要）で Lambda との相性が抜群。
    - 読み書きが発生した分だけの従量課金。
    - スケーラビリティが高く、不定期な大量レポート処理に適している。

### 2.3 API インターフェース
- **決定:** Amazon API Gateway (HTTP API)
- **理由:**
    - REST API よりも安価（約1/3のコスト）で高速。
    - CORS 設定が容易。
    - 今回のシンプルな Lambda プロキシ構成に最適。
    - 開発時のデバッグ（REST Client からの直接呼び出し）が容易。

## 3. 実装詳細

### 3.1 データモデリング (DynamoDB)
一意のキーが存在しない明細データを扱うため、以下のキー設計を採用：
- **Partition Key (PK):** `settlementId` (ファイル内で共通)
- **Sort Key (SK):** `lineId` (パース時に生成する連番: L1, L2...)
- **理由:** `orderId` や `postedDateTime` の組み合わせでも一意性が保証されないケースに対応し、かつ確実に全レコードを保存するため。

### 3.2 ファイル処理
- タブ区切り (TSV) 形式に対応。
- 2行目のサマリーレコード（ヘッダー情報のみの行）をスキップし、3行目以降の明細のみを抽出。
- `busboy` ライブラリを使用して `multipart/form-data` を解析。

## 4. 考慮した代替案

### 4.1 S3 トリガー方式
- **検討理由:** 大容量ファイルへの対応。
- **却下理由:** 現状のファイルサイズ（1MB未満）では API Gateway で十分であり、同期的なレスポンス（デバッグのしやすさ）を優先した。将来的にファイルが巨大化する場合は、S3 経由への移行を検討する。

### 4.2 Java Lambda (SnapStart)
- **検討理由:** 既存コードの再利用。
- **却下理由:** SnapStart を使っても TypeScript の軽量さには及ばず、また今後のメンテナンス性を考慮してモダンな TypeScript 環境への移行を選択した。
